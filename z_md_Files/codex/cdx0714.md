# cdx0714 작업 기록

## 2026-07-14 CastleChat 버그 수정

1. 초대하기가 다시 작동하지 않던 문제.
   - 원인: 프론트의 `emitWsInviteMember`가 단순 전송만 하고 성공/실패 응답을 기다리지 않았다. 서버에서 실패해도 프론트는 로컬 멤버 목록을 먼저 만질 수 있는 구조였다.
   - 수정: `emitWsInviteMember`를 `emitWsRequest` 기반으로 바꿨다. 성공 응답은 `ROOM_MEMBER_INVITED`, 실패 응답은 `INVITE_MEMBER_FAIL`, `WS_MESSAGE_FAIL`로 받는다.
   - 수정: `ChatBox`의 초대 로직은 서버 성공 응답을 받은 뒤에만 로컬 멤버 목록을 갱신한다.

2. 답장 target 메시지가 exit 후 다시 enter하면 안 보이던 문제.
   - 원인: `replyToMessageId`는 내려오지만, 화면에 로딩된 메시지 목록 안에서 target 메시지를 찾지 못하면 답장 preview 자체가 렌더링되지 않았다.
   - 수정: target 메시지가 현재 로딩 범위에 없으면 `이전 답장 메시지` fallback preview를 보여주게 했다.
   - target 메시지가 현재 목록에 있으면 기존처럼 클릭 시 해당 메시지로 이동한다.

3. leftRoom 시 채팅방 목록에서 방이 실시간으로 사라지지 않던 문제.
   - 원인: `ChatList`가 `LEFT_ROOM`, `LEFT_ROOM_OK` 이벤트를 목록 제거 이벤트로 처리하지 않았다.
   - 수정: 나 자신이 나간 방이면 React Query 캐시와 화면 목록에서 즉시 제거하도록 했다.

4. 방 설정 변경 성공 안내.
   - 수정: 방 설정 저장 성공 시 채팅방 메뉴 내부에 `적용되었습니다.` 토스트를 1.8초 동안 표시한다.
   - alert가 아니라 작은 안내창으로 처리해서 흐름을 막지 않게 했다.

5. left한 유저의 과거 메시지가 `알 수 없음`과 기본 이미지로 보이던 문제.
   - 원인: 메시지 렌더링이 현재 active 멤버 목록인 `memberMap`에 의존했다. 유저가 방을 나가면 `room_members.member_status`가 `LEFT`가 되고 active 멤버 목록에서 빠진다. 그 결과 과거 메시지의 `senderPublicId`는 있어도 닉네임과 프로필을 찾지 못해 fallback인 `알 수 없음`, 기본 이미지로 떨어졌다.
   - 정책: 단순 left는 탈퇴가 아니므로 과거 메시지의 작성자 정보는 유지되어야 한다.
   - 수정: `ChatMessageViewResponseDTO`에 `senderNickname`, `senderProfileImg`를 추가했다.
   - 수정: 이전 메시지 조회 SQL에서 `chat_messages.sender_id -> users.user_id` 조인으로 sender 닉네임과 프로필 이미지를 함께 내려주게 했다.
   - 수정: 프론트는 active 멤버 목록에서 sender를 찾지 못하면 메시지 자체의 `senderNickname`, `senderProfileImg`를 fallback으로 사용한다.
   - 결과: 유저가 방을 나가도 `users` row가 남아 있는 한 과거 메시지의 작성자 정보가 유지된다.

6. 사진 메시지의 초록색 테두리/배경 제거.
   - 원인: 이미지 첨부만 있는 메시지도 일반 말풍선 `.messageWrap` 배경을 그대로 탔다.
   - 수정: 이미지 첨부만 있고 텍스트/답장이 없는 메시지에 `imageOnlyMessage` 클래스를 붙였다.
   - 수정: 해당 클래스는 padding, background, shadow를 제거해서 사진만 보이게 했다.

7. 공유하기가 alert만 뜨던 문제.
   - 수정: 우클릭 메뉴의 공유하기에서 target roomId를 입력받아 기존 `SEND_MESSAGE` 흐름으로 `[공유] ...` 텍스트 메시지를 보낸다.
   - 현재는 최소 동작 연결이며, 방 선택 UI는 추후 개선 대상으로 남긴다.

8. 채팅방 메뉴 위치 문제.
   - 원인: 도킹 모드 CSS에서 `.roomSidePanel`을 채팅방 목록 쪽으로 음수 이동시키고 있었다.
   - 수정: 도킹 모드에서도 채팅방 패널의 왼쪽 끝에서 슬라이드되도록 `left: 0`, `top: 46px`로 조정했다.

## 2026-07-14 추가 수정

1. 채팅방 메뉴를 슬라이드 방식에서 open/close 모달 방식으로 변경했다.
   - 기존 `.roomSidePanel`은 `transform: translateX(-100%)`와 transition으로 왼쪽에서 밀려오는 구조였다.
   - 요청대로 같은 위치에서 열리고 닫히는 모달처럼 보이도록 `display: none/flex` 방식으로 바꿨다.
   - 위치는 현재 채팅방 메뉴 위치를 유지했다.

2. 초대하기 실패 원인을 수정했다.
   - 로그 원인: `Data truncated for column 'message_type'`.
   - 실제 원인: 초대 성공 후 room feed를 `chat_messages`에 저장하면서 `message_type = 'SYSTEM'`을 insert했다. 그런데 현재 DB의 `chat_messages.message_type` enum에는 `SYSTEM`이 없다.
   - 수정: feed 저장 시 DB에는 유효한 enum 값인 `TEXT`로 저장하고, `message_text` 앞에 `[ROOM_FEED]` prefix를 붙였다.
   - 수정: 이전 메시지 조회 시 `[ROOM_FEED]` prefix가 있으면 프론트에는 `messageType = SYSTEM`으로 내려주고, text에서는 prefix를 제거한다.

3. 특정 메시지 공지하기 실패 원인을 수정했다.
   - 공지 적용 자체가 아니라, 공지 적용 후 생성하는 room feed 저장에서 초대와 같은 `SYSTEM` insert 오류가 발생했다.
   - 2번과 같은 feed 저장 구조 변경으로 같이 해결했다.

4. 멤버 권한 변경 실패 원인을 수정했다.
   - 권한 변경 서비스는 update 후 `createRoomFeed`를 호출한다.
   - 첨부 로그에는 최종 cause가 생략되어 있었지만, 코드 흐름상 초대/공지와 동일하게 feed 저장의 `SYSTEM` enum 문제가 권한 변경도 막을 수 있었다.
   - feed 저장 구조를 `TEXT + [ROOM_FEED] prefix`로 바꿔 권한 변경 feed도 DB enum 오류 없이 저장되게 했다.

5. 채팅방 목록 쪽 보정.
   - feed row가 DB상 `TEXT`로 저장되기 때문에 unread 계산에 끼어들 수 있었다.
   - 채팅방 목록 unread 계산에서는 `[ROOM_FEED]` 메시지를 제외했다.
   - 마지막 메시지 preview에는 `[ROOM_FEED]` prefix가 보이지 않도록 제거했다.

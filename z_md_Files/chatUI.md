< 채팅 UI 구조 결정 기록 (카톡식 멀티윈도우 vs 디스코드식 싱글뷰) >

작성 : 2026-07-11. 취업(백엔드/풀스택) 면접 관점까지 포함해서 결정한 기록이다.

===========================================================================================

A. 고민 배경

- 현재 castleChat은 카톡 PC처럼 "채팅방 하나당 창(플로팅 패널) 하나"를 띄우고, 여러 방을 동시에 띄울 수 있는 구조다.
- 그런데 이건 채팅"웹"이라, 디스코드/슬랙처럼 "좌측 리스트 + 한 화면에 방 하나" 구조가 웹 관례 아닌가 하는 고민.
- 인턴 시절 사내 채팅앱은 카톡 PC식이었지만, 걔는 웹이 아니라 별도 로컬 exe였다.

B. 먼저 짚을 사실 : 현재 구조는 백엔드 관점에서 이미 올바르다

코드 확인 결과(wsClient.js), "창"은 브라우저 새 창(window.open)이 아니라 SPA 내부의 플로팅 패널이고,
WebSocket은 모듈 스코프에 딱 1개만 유지하면서 roomHandlers로 방별 이벤트를 라우팅(멀티플렉싱)한다.

즉 "창을 10개 띄워도 서버 연결은 1개"다. wsgate 입장에서 연결 수는 유저 수와 같고, UI가 몇 창이든 서버 비용이 안 늘어난다.
이건 prpr 1(대용량 트래픽) 관점에서 정확한 설계고, 면접에서 자신 있게 말할 수 있는 부분이다.
--> 따라서 이번 고민은 순수하게 "화면을 어떻게 보여줄 것인가"의 문제지, 아키텍처를 갈아엎는 문제가 아니다.

C. 두 안 비교

C-1. 카톡 PC식 (현재. 방마다 플로팅 창, 동시 다개)

장점
- 여러 방을 동시에 보면서 왔다갔다 하는 멀티태스킹이 편하다.
- 국내 사용자에게 익숙하다. 사내 메신저(인턴 경험)도 이 구조였다.
- 이미 구현되어 있다.

단점
- 웹 제품 관례가 아니다. 디스코드/슬랙/팀즈/텔레그램웹/왓츠앱웹 등 메이저 웹 채팅은 전부 싱글뷰다.
  이유가 있다 : 웹은 "브라우저 창" 안에서 돌아서 화면이 제한적이고, 모바일 브라우저에서는 플로팅 멀티윈도우가 성립 자체가 안 된다.
- 반응형(모바일) 질문이 들어오면 답이 궁색해진다. "모바일에선 어떻게 되나요?"에 대한 답이 없다.
- 창 z-index/포커스/위치 관리 등 FE 복잡도가 계속 는다.

C-2. 디스코드식 (좌측 방 리스트 + 메인 뷰에 방 하나)

장점
- 웹 채팅의 사실상 표준. 면접관이 데모를 열었을 때 "웹 제품답다"는 첫인상을 준다.
- 모바일 반응형이 자연스럽다 (리스트 <-> 방 화면 전환).
- 화면이 단순해져서 unread 뱃지/정렬 같은 UX 기능이 리스트에 집중되고 잘 보인다.

단점
- 여러 방 동시 보기가 안 된다 (카톡식 멀티태스킹 상실).
- 지금 구조에서 레이아웃 개편 작업이 필요하다.

D. 면접 유불리 판단

1. 백엔드 지원자 기준 : UI 스타일 자체는 합격/불합격을 가르지 않는다. 면접관이 보는 건
   "왜 그렇게 만들었는지 설명할 수 있는가"와 UI 뒤의 설계(연결 1개 멀티플렉싱, read position 비동기 처리)다.
2. 하지만 데모 첫인상은 실재한다. 웹 포트폴리오를 열었는데 웹 관례와 다른 화면이 나오면
   "왜 웹인데 데스크톱 앱 UI죠?"라는 질문이 먼저 나오고, 방어부터 시작하는 면접이 된다.
   디스코드식이면 그 질문 자체가 안 나오고, 바로 아키텍처 얘기로 들어갈 수 있다.
3. 풀스택 포지션이면 더 기운다 : "플랫폼 관례를 알고 따랐는가"가 곧 제품 감각 평가라서 디스코드식이 유리하다.
4. 카톡식의 멀티태스킹 가치는 진짜지만, 그건 "기본 레이아웃"이 아니라 "부가 기능"으로 넣을 때 제일 빛난다 (E 참고).

E. 결론 : 디스코드식을 기본으로, 기존 플로팅 창은 "팝아웃" 기능으로 재활용

1. 기본 화면은 디스코드식으로 간다. 좌측 방 리스트(unread 뱃지, 최근 메시지 미리보기) + 메인 뷰에 활성 방 하나.
2. 이미 만들어 둔 플로팅 창 컴포넌트는 버리지 않는다. 방 헤더에 "팝아웃" 버튼을 두고,
   누르면 그 방이 플로팅 창으로 분리되어 다른 방을 보면서 동시에 대화할 수 있게 한다. (슬랙의 창 분리와 같은 발상)
3. 이러면 면접 스토리가 제일 강해진다 :
   "웹 관례(싱글뷰)를 기본으로 따르되, 멀티태스킹 니즈는 팝아웃으로 해결했다.
    어느 쪽이든 WS 연결은 1개로 멀티플렉싱되므로 서버 비용은 동일하다."
   --> 관례 준수 + 트레이드오프 인지 + 백엔드 근거, 세 가지가 한 문장에 들어간다.
4. 작업 순서는 레이아웃 전환이 먼저, 팝아웃은 그 다음(선택). 팝아웃은 안 해도 감점 없다. 있으면 차별화 포인트.

F. 면접 예상 문답 (준비용)

Q. 왜 디스코드식 UI를 택했나?
A. 웹은 데스크톱 네이티브와 달리 브라우저 창/모바일 화면이라는 제약이 있어서, 메이저 웹 채팅 제품이 전부 싱글뷰 관례를 쓴다.
   그 관례를 기본으로 따르고, 카톡식 멀티태스킹 니즈는 팝아웃 기능으로 흡수했다.

Q. 창을 여러 개 띄우면 서버 부담이 늘지 않나?
A. 안 는다. 창이 몇 개든 브라우저 탭당 WebSocket 연결은 1개고, 방별 이벤트는 클라이언트에서 핸들러로 라우팅한다.
   게이트웨이의 최고 희소자원이 동시 연결 수라서, 연결 수를 유저 수 이상으로 안 늘리는 게 설계 원칙이었다.

Q. 여러 방이 동시에 보이면 읽음 처리가 복잡해지지 않나?
A. 읽음 위치는 동기 DB CRUD가 아니라 Redis 최신값 갱신 + dirty flush worker의 batch 반영이라,
   어느 창에서 읽든 "최신 read position 하나"로 병합된다. UI 구조와 무관하게 서버 부하가 read 횟수에 비례하지 않는다.

H. cdx 최종 답변

결론부터 말하면, 둘 중 하나만 선택해야 한다면 웹 포트폴리오의 기본 UI는 디스코드식 싱글뷰가 더 유리하다.
다만 현재 구현한 카톡 PC식 멀티윈도우를 폐기할 필요는 없다. 가장 좋은 완성형은 "디스코드식 싱글뷰를 기본으로 두고, 원하는 방만 팝아웃하는 하이브리드 구조"다.

이 판단의 핵심은 다음과 같다.

1. 인사 담당자와 비개발 면접관에게는 첫 화면의 정돈감과 사용법의 직관성이 중요하다. 싱글뷰는 화면 크기가 달라도 레이아웃이 쉽게 무너지지 않고, 별도 설명 없이도 사용법을 이해하기 쉽다.
2. 풀스택 면접에서는 웹의 화면 제약과 반응형 관례를 고려했다는 점이 제품 감각으로 평가될 수 있다. 이 부분은 싱글뷰가 더 안전하다.
3. 백엔드 면접에서는 UI 형태보다 하나의 WebSocket 연결에서 여러 방의 이벤트를 어떻게 분기하고, 읽음 위치와 unread count를 어떻게 처리했는지가 훨씬 중요하다. 따라서 싱글뷰로 바꿔도 지금까지 만든 실시간 아키텍처의 가치는 사라지지 않는다.
4. 카톡식 멀티윈도우의 동시 대화 장점은 분명하다. 이를 기본 화면으로 강제하기보다 사용자가 선택하는 팝아웃 기능으로 제공하면 차별화 요소가 된다.

면접 평가만 놓고 보면 우선순위는 "안정성과 완성도 > 선택 근거 > UI 방식 자체"다. 디스코드식을 선택했더라도 버그가 많으면 불리하고, 카톡식을 선택했더라도 데스크톱 중심의 멀티태스킹 도구라는 근거와 안정적인 동작을 보여주면 충분히 방어할 수 있다.

현재 프로젝트의 현실적인 적용 순서는 다음이 적절하다.

1. 채팅 목록과 활성 채팅방 하나를 보여주는 싱글뷰를 기본 화면으로 만든다.
2. 기존 ChatBox와 다중 창 상태 관리는 제거하지 않고 팝아웃 기능으로 재사용한다.
3. 화면이 좁아지면 팝아웃을 제한하고 싱글뷰만 사용하게 한다.
4. 포트폴리오 시연에서는 기본 싱글뷰로 안정적인 첫인상을 보여준 뒤, 팝아웃으로 여러 방을 동시에 처리하는 차별화 기능을 보여준다.

면접에서는 다음과 같이 설명하면 된다.

"웹의 반응형 환경과 익숙한 사용 흐름을 고려해 싱글뷰를 기본으로 선택했습니다. 다만 여러 대화를 동시에 확인하려는 사용자를 위해 기존 멀티윈도우 구현을 팝아웃 기능으로 유지했습니다. 두 UI 모두 WebSocket 연결을 방마다 새로 만들지 않고 하나의 연결에서 방별 이벤트를 라우팅하므로, 사용자 편의 기능이 서버 연결 수 증가로 이어지지 않도록 설계했습니다."

따라서 최종 추천은 "디스코드식 기본 화면 + 카톡 PC식 팝아웃"이다. 일정 때문에 당장 둘을 모두 완성하기 어렵다면, 면접용 기본 화면은 디스코드식 싱글뷰를 먼저 완성하는 편이 합격 관점에서 더 안전하다.

I. 추가 논점 : 1차 심사자가 "비전공 인사팀"이라는 현실 (2026-07-11)

I-1. 이 사실은 결론을 바꾸지 않고, 오히려 강화한다

비전공자가 프로젝트에서 느끼는 건 세 가지뿐이다. "이쁘다/정돈됐다", "뭘 눌러야 할지 바로 알겠다", "안 깨진다".

- 디스코드식 싱글뷰는 학습 비용이 0이다. 열면 방 목록이 보이고, 누르면 대화가 보인다. 컴맹도 헤맬 수가 없다.
- 카톡식 플로팅 창은 "창을 여러 개 띄울 수 있다"는 걸 알아야 가치가 드러나는데, 비전공자는 그걸 발견하지 못한다.
  발견해도 창이 겹치고 흩어진 화면은 스크린샷/첫인상에서 "정돈 안 됨"으로 읽힌다.
- 스크린샷 한 장으로 승부가 나는 서류 단계에서, 싱글뷰는 한 장에 제품 전체가 담긴다. 플로팅 창은 안 담긴다.

I-2. 진짜 중요한 것 : 인사팀은 UI "방식"이 아니라 "접근 장벽"과 "때깔"로 거른다

인사팀 직원은 코드를 안 본다. 서류의 스크린샷을 보고, 잘해야 배포 링크를 30초 클릭해본다. 그 30초에서 승부가 난다.

1. 배포 링크 + 원클릭 체험 계정이 최우선이다. 회원가입 화면이 나오는 순간 인사팀은 닫는다.
   "체험하기" 버튼 하나로 자동 로그인되고, 미리 채워진 대화방/친구/메시지가 보여야 한다. (시연용 시드 데이터)
2. 첫 화면 비주얼 완성도가 UI 구조 선택보다 훨씬 크게 작용한다. 같은 싱글뷰라도 여백/폰트/색만 정돈돼도 "제품 같다"가 된다.
3. AI 메시지 추천 기능이 비전공자에게 제일 큰 "와" 포인트다. kafka/redis는 인사팀에게 안 보이지만 AI 버튼은 보인다.
   시연 동선에서 잘 보이는 위치에 배치할 것.
4. 포트폴리오 문서에는 GIF(움직이는 시연 장면)를 넣는다. 비전공자는 글을 안 읽고 그림을 본다.

I-3. 정리

- UI 구조 결정은 그대로 : 디스코드식 기본 + 팝아웃. (인사팀 관점에서도, 실무자 관점에서도 같은 답)
- 단, 인사팀 관문을 위해 FE 개편 시 함께 챙길 것 : 체험 계정 원클릭 진입, 시연용 시드 데이터, 첫 화면 폴리싱, AI 추천 버튼 노출.

단, 시연 일정이 임박했고 현재 멀티윈도우가 안정적으로 동작한다면 전면 재작성은 하지 않는다. 기존 상태 관리와 ChatBox를 유지한 채 활성 방 하나를 본문에 배치하는 포커스 모드를 먼저 추가하고, 검증이 끝난 뒤 그 모드를 기본값으로 전환하는 방식이 안전하다. 완성된 비표준 UI가 깨진 표준 UI보다 면접에서 훨씬 낫다.
